# Contributing
## Before you open a pull request
- Keep the change scoped and describe the reason for it clearly.
- Run the tests, linting, and type checks that apply to the repository you changed.
- Update docs, examples, or contracts when the change affects user-visible behavior.

## Pull request expectations
- Open pull requests against the default branch unless the repository documents a different release flow.
- Include a short summary, how you validated the change, and any rollout or migration notes.
- Call out security, privacy, compliance, or deployment impact explicitly.
- Expect review from the repository owners defined in `CODEOWNERS`.

## Commit and review hygiene
- Prefer small, reviewable commits and pull requests.
- Do not merge with unresolved review threads.
- Do not bypass required checks or owner review without a documented reason.

## Security
Do not open public issues for security vulnerabilities. Follow `SECURITY.md` instead.
